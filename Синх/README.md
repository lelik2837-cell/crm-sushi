# Синхронизация выручки, времени ожидания и заказов: papasushi.goulash.tech → crmpapa.ru

Что делает скрипт `revenue_sync.py`:

1. Логинится в личный кабинет `papasushi.goulash.tech` (та же форма, что и обычный вход в браузере) —
   один раз за запуск, даже если филиалов несколько.
2. Для каждого настроенного филиала дёргает внутренний JSON-эндпоинт
   `GET /dashboad/api/header?departmentId=...` — это тот же запрос, который сама страница дашборда
   выполняет каждую минуту для обновления шапки (найден через вкладку Network в DevTools).
3. Берёт из ответа поле `total_summ_clear` — это и есть выручка — плюс `count_orders` (число заказов).
4. Отправляет их JSON-POST-запросом на webhook-URL этого филиала на crmpapa.ru.
5. **(Если настроено — см. раздел про `GOULASH_WAITTIME_SYNC_MAP`)** из того же ответа достаёт
   время ожидания — запись `{"name": "Обещаем гостю", "values": [...]}` в `department_info.messages` —
   и отправляет отдельным запросом на второй вебхук того же филиала (свой токен). Выручка и время
   ожидания идут в разные вебхуки на crmpapa.ru, но забираются одним и тем же запросом к goulash —
   лишней нагрузки на goulash это не добавляет.
6. **(Если настроено — см. раздел «Заказы «Отчёт по заказам»»)** тем же логином скачивает CSV-выгрузку
   «Отчёт по заказам» за последние несколько дней и пересылает её на отдельный вебхук заказов —
   вместо того, чтобы скачивать и загружать этот файл в CRM вручную.

## 1. Открыть в VS Code

Откройте папку с этими файлами в VS Code (File → Open Folder).
Файлы:

- `revenue_sync.py` — сам скрипт
- `requirements.txt` — зависимости
- `.env.example` — шаблон переменных окружения
- `Dockerfile` — для запуска на Railway (см. раздел 5)

## 2. Установка

В терминале VS Code:

```bash
python3 -m venv venv
source venv/bin/activate        # на Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## 3. Настройка (несколько филиалов)

```bash
cp .env.example .env
```

Откройте `.env` и заполните:

- `GOULASH_USERNAME` / `GOULASH_PASSWORD` — логин и пароль от papasushi.goulash.tech.
  Один аккаунт видит все филиалы, отдельный логин на каждый филиал не нужен.
- `GOULASH_SYNC_MAP` — по одной паре `department_id=webhook_url` на каждый филиал, через `;`.

**Как узнать `department_id` каждого филиала:** зайдите в личный кабинет goulash в браузере,
откройте вкладку **Network** в DevTools, переключите филиал в интерфейсе — увидите запрос
`GET /dashboad/api/header?departmentId=XXX`, вот это число и есть нужный id.

**Как узнать `webhook_url` каждого филиала:** на crmpapa.ru — Настройки → Интеграции →
«Синхронизация выручки (вебхук)» → создать токен для этого филиала → скопировать готовый URL.

Пример для 8 филиалов (просто добавьте по паре на каждый через `;`, без пробелов вокруг `;`):

```
GOULASH_SYNC_MAP=162=https://crmpapa.ru/api/revenue-webhook/tok1;165=https://crmpapa.ru/api/revenue-webhook/tok2;170=https://crmpapa.ru/api/revenue-webhook/tok3;...
```

### Время ожидания «Обещаем гостю» (необязательно)

Если нужно ещё и время ожидания — заполните `GOULASH_WAITTIME_SYNC_MAP` **теми же** `department_id`,
что и в `GOULASH_SYNC_MAP`, но с другими URL — токенами из Настройки → API →
«Синхронизация времени ожидания» (это отдельный раздел от выручки, отдельные токены):

```
GOULASH_WAITTIME_SYNC_MAP=162=https://crmpapa.ru/api/waittime-webhook/tokA;165=https://crmpapa.ru/api/waittime-webhook/tokB;...
```

Если department_id есть в `GOULASH_SYNC_MAP`, но отсутствует в `GOULASH_WAITTIME_SYNC_MAP` —
для этого филиала просто продолжит отправляться только выручка, ничего не сломается.
Если переменную не задавать вообще — всё работает как раньше, без времени ожидания.

### Заказы «Отчёт по заказам» (необязательно)

До этого заказы из отчёта скачивались вручную (CSV-файл со страницы `receipts/receipts/index`)
и загружались в CRM через Настройки → «Отчёт по заказам» → «Импорт». Если задать переменные
ниже — скрипт сам логинится, скачивает этот же CSV за последние несколько дней и отправляет
его на CRM тем же способом, каждый цикл опроса, без ручной выгрузки/загрузки файла.

Задайте:

- `GOULASH_ORDERS_QUERY_TEMPLATE` — строка фильтра страницы отчёта. В `.env.example` уже
  подставлено рабочее значение (те же филиалы/статус заказа/набор колонок, что были
  настроены на странице изначально) — трогать не нужно, если набор филиалов на goulash
  не менялся. Если понадобится обновить: откройте страницу отчёта в браузере с нужными
  фильтрами, скопируйте всё, что в адресной строке идёт **после** `?`, и замените в этой
  строке конкретные даты (`ReceiptsAll[date_s]=...` и `ReceiptsAll[date_do]=...`) на
  `{date_from}` и `{date_to}` буквально — их подставляет сам скрипт перед каждым запросом.
  **Важно:** если в адресной строке окажется `ReceiptsAll[isCurrentDate]=0` **и следом**
  `ReceiptsAll[isCurrentDate]=1` (так бывает, если в форме отчёта отмечен чекбокс
  «текущая дата») — оставьте только `=0`, иначе Гуляш полностью игнорирует
  `date_s`/`date_do` и отдаёт только сегодняшний день (найдено и исправлено при
  первом реальном запуске — см. plan.md, п.176).
- `CRMPAPA_ORDERS_WEBHOOK_URL` — Настройки → API → «Синхронизация заказов (вебхук)» на
  crmpapa.ru → создать токен → скопировать готовый URL. Токен один сразу на все филиалы —
  каждый заказ и так несёт своё подразделение внутри самой выгрузки.
- `ORDERS_LOOKBACK_DAYS` (необязательно, по умолчанию `3`) — за сколько последних дней
  запрашивать заказы при каждом запуске. Специально с запасом больше одного дня — чтобы
  подхватывать заказы, которые "дозрели" (сменили сумму/время доставки) уже после первой
  загрузки. Это безопасно: повторная загрузка уже известного заказа не создаёт дубль на
  crmpapa.ru, а обновляет его строку.

Если на goulash появится подразделение, которого раньше не было в выгрузках — заказ всё
равно загрузится (просто без привязки к филиалу), а сопоставить его с нужным филиалом
можно потом на странице «Филиалы» на crmpapa.ru, как и раньше при ручной загрузке.

Как скрипт получает именно файл, а не страницу: страница `receipts/receipts/index` при
обычном заходе отдаёт HTML (саму страницу с таблицей) — сам файл отдаёт этот же URL
с дополнительным параметром `is_excel=true` (это разобрано из JS кнопки «Выгрузить в Excel»
прямо на странице отчёта — она берёт текущий фильтр, добавляет этот параметр и переходит
по тому же адресу). `fetch_orders_csv()` добавляет его сама, отдельно настраивать не нужно.
На всякий случай (если сайт когда-нибудь поменяет вёрстку/JS) скрипт всё равно проверяет
ответ (`_looks_like_orders_csv()` ищет колонку «Номер заказа») и, если получил не то, пишет
в лог понятную ошибку с началом ответа вместо того, чтобы упасть молча или тихо ничего не
загрузить — если увидите такую ошибку, пришлите эту строку из лога.

Скрипт сейчас читает `.env` не автоматически — либо экспортируйте переменные в шелл, либо добавьте
в начало `revenue_sync.py` пакет `python-dotenv` (`pip install python-dotenv` и
`from dotenv import load_dotenv; load_dotenv()`), если хотите подхватывать `.env` из файла.

Быстрый вариант без dotenv — экспортировать переменные прямо в терминале:

```bash
export GOULASH_USERNAME=ваш_логин
export GOULASH_PASSWORD=ваш_пароль
export GOULASH_SYNC_MAP="162=https://crmpapa.ru/api/revenue-webhook/tok1;165=https://crmpapa.ru/api/revenue-webhook/tok2"
```

Если филиал всего один — можно вместо `GOULASH_SYNC_MAP` задать просто
`GOULASH_DEPARTMENT_ID` и `CRMPAPA_WEBHOOK_URL` (старый однофилиальный режим, оставлен для
совместимости).

## 4. Запуск

Один раз (проверить, что всё работает):

```bash
python revenue_sync.py --once
```

Постоянно, с опросом каждые 10 минут (значение берётся из `POLL_INTERVAL_SECONDS`):

```bash
python revenue_sync.py
```

Либо, если хотите не держать процесс запущенным вечно, а гонять по cron — используйте
`--once` в cron-задаче:

```
*/10 * * * * cd /path/to/project && venv/bin/python revenue_sync.py --once >> sync.log 2>&1
```

## 5. Постоянная работа 24/7 — Docker Compose на сервере CRM (актуальная схема)

**УСТАРЕЛО:** раньше здесь была инструкция для отдельного сервиса на Railway — Railway для
этого проекта больше не используется вообще (ни для самой CRM, ни для этого скрипта, с
2026-07-19). Сейчас `Синх` работает как ещё один контейнер `docker compose` рядом с самой
CRM, на одном сервере.

Устройство (сервер `109.73.195.176`, папка `/opt`):

- `/opt/docker-compose.yml` — три сервиса: `web` (сама CRM, порт 8080), `worker` (этот
  скрипт, `build: /opt/crm-sushi/Синх`), `caddy` (reverse-proxy, TLS для `crmpapa.ru`).
- `/opt/env/worker.env` — переменные окружения именно этого скрипта (`GOULASH_USERNAME`,
  `GOULASH_PASSWORD`, `GOULASH_SYNC_MAP`, `GOULASH_WAITTIME_SYNC_MAP`,
  `GOULASH_ORDERS_QUERY_TEMPLATE`, `CRMPAPA_ORDERS_WEBHOOK_URL`, `POLL_INTERVAL_SECONDS`,
  `TZ=Asia/Novosibirsk`) — правится напрямую на сервере, в `.env`/`.env.example` в git не
  попадает (там только пример/шаблон для справки).
- `/opt/deploy.sh` — вызывается GitHub Actions при пуше в `main` (см. `deploy-workflow` в
  памяти проекта): `git pull` в `/opt/crm-sushi`, затем `docker compose build && docker
  compose up -d` — пересобирает и перезапускает **все три** контейнера сразу, включая
  `worker`, так что изменения в `Синх/revenue_sync.py` доезжают автоматически при обычном
  деплое CRM, отдельно ничего не нужно.
- Если поменялась только переменная окружения (например обновили токен вебхука), без
  изменений в коде — код-деплой не нужен, правьте `/opt/env/worker.env` прямо на сервере и
  перезапустите только этот контейнер: `cd /opt && docker compose up -d worker` (пересоздаст
  только `worker`, `web`/`caddy` не тронет).
- Логи: `docker logs opt-worker-1` (там будут строки "Успешно авторизовались", "Отправлено
  на crmpapa.ru (филиал ...)", "Заказы ...: импортировано ..." и т.д.) — либо разделы «Лог
  входящих запросов» в Настройки → API на crmpapa.ru (отдельно для выручки, времени ожидания
  и заказов).

Секреты (логин/пароль от goulash, URL вебхуков) хранятся только в переменных окружения
Railway — они не попадают в git и не видны в коде.

## Возможные проблемы

- **Логин не проходит** — сайт на Yii2, скрипт пытается найти CSRF-токен автоматически
  (`_csrf-frontend` / `_csrf` / мета-тег). Если разметка формы изменится, поправьте
  `get_csrf_token()`.
- **Ответ не JSON / сессия истекла** — `fetch_department_data()` кидает понятную ошибку в этом случае;
  скрипт в режиме постоянного опроса просто залогинится заново на следующей итерации.
- **Один филиал упал, остальные должны продолжать** — `run_once()` ловит ошибку на каждом
  филиале отдельно (пишет в лог и идёт дальше), сбой одного department_id не останавливает
  синхронизацию остальных.
- **Двухфакторная авторизация или вход по отпечатку/штрихкоду** — на странице логина видна
  фраза "Можно авторизоваться по отпечатку пальца или по штрих коду". Если на вашем аккаунте
  включена такая защита, обычный логин/пароль через форму может не сработать — тогда нужен
  отдельный технический пользователь с простой авторизацией, если система это поддерживает.
- **Время ожидания не приходит / всегда пустое** — точный формат `messages[].values` в ответе
  goulash при непустом значении не был подтверждён на реальных данных при написании скрипта.
  `_extract_message_value()` умеет разбирать несколько вероятных вариантов (число, `{"value": ...}`,
  `{"minutes": ...}`, число строкой) и пишет в лог `"Не удалось разобрать values для ..."`, если
  формат окажется другим — это не ошибка запроса, просто нужно поправить разбор под реальный формат.
  Пришлите такую строку из лога (или пример JSON-ответа) — поправим `_extract_message_value()`.
- **Заказы: «Ответ не похож на CSV-выгрузку...» в логе** — не ошибка запроса как таковая, а то,
  что `fetch_orders_csv()` честно проверил ответ и не нашёл в нём ожидаемую колонку «Номер заказа».
  Обычно значит, что истекла сессия (редко — скрипт логинится заново каждый запуск), либо сайт
  поменял механизм экспорта (сейчас это параметр `is_excel=true` к тому же URL — см. выше). Пришлите
  начало ответа, которое попадёт в эту строку лога — поправим `fetch_orders_csv()`.
- **Заказы приходят, но у части филиал не проставлен** — на crmpapa.ru заказ всё равно загрузится
  (просто без филиала), нужно один раз сопоставить название подразделения с филиалом на странице
  «Филиалы» — как и раньше при ручной загрузке того же CSV, дальше сопоставление запоминается.
