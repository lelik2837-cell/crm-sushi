### 27. Исправление ложного матчинга контрагентов

**Проблема:** ООО "КОМПАНИЯ ВОДА" распознавалась как СОУСПАК, ООО
- Причина 1: текст для поиска включал `description` (назначение платежа) — слово «ООО» из ключевых слов СОУСПАК совпадало с «ООО» в поле description

- Матчинг теперь только против поля `counterparty`, не `description + counterparty`
- Если ключевые слова не заданы вручную (`keywords = ''`) → используется имя контрагента целиком как одна фраза (не разбивается по запятой)
- Если ключевые слова заданы (`keywords` заполнен) → разбиваются по запятой как прежде

### 78. Отчёт по выручке — сводная таблица с планом

**Новая таблица `revenue_plan (branch_id, date, amount, PRIMARY KEY (branch_id, date))`**

**Фильтры:**
- Вместо выбора периода (7 дней / месяц / 30 дней) — явный выбор дат от/до
- По умолчанию — текущий месяц (включая будущие даты)
- Flatpickr range picker: один календарь, выбор диапазона кликом на начало и конец
- Кнопки навигации `‹ Июнь 2026 ›` — переключение на предыдущий/следующий месяц с автоприменением

**Сводная таблица:**
- Строки = даты, столбцы = филиалы × {Выручка | План | Отклонение} + Итого
- Ячейки плана редактируются кликом (inline editing), сохраняются через `POST /reports/revenue-plan/save`
- Отклонение = факт − план: зелёный ≥100%, оранжевый ≥90%, красный <90%
- Если выручка отсутствует (actual=0) → отклонение показывается как «—»
- Отклонение за период вынесено в строку с названием филиала в шапке

**Авто-план:**
- Вводишь общий план за период → система берёт выручку прошлого года за тот же период
- Смещение 364 дня (52 недели) сохраняет день недели → пятница 2026 = пятница 2025
- Коэффициент роста = total_plan / last_year_total → распределяет пропорционально по дням и филиалам
- `POST /reports/revenue-plan/auto` — возвращает обновлённый план, JS перерисовывает ячейки

**Карточки над таблицей:** Факт за период / План за период / % выполнения

**UI таблицы:**
- Шапка: светло-голубой фон, названия филиалов 14px bold
- Чередующиеся фоны колонок: нечётные — светло-жёлтый (`#fefce8`), чётные — белый
- При наведении на строку — полупрозрачный фиолетовый оверлей (`box-shadow: inset`)

## Глобальные стандарты UI (обязательно везде)

### Формат даты
- **Везде отображать даты в формате `DD-MM-YYYY`** (например `22-06-2026`)
- Хранение в БД: ISO `YYYY-MM-DD`; `<input type="date">` тоже принимает `YYYY-MM-DD`
- Конвертация для отображения: Jinja2-фильтр `datefmt` в `app.py` (`{{ val | datefmt }}`)

### Выпадающий список филиалов
- **Везде использовать мультивыбор с галочками и группами** (Bootstrap dropdown + checkboxes)
- Паттерн: кнопка-dropdown показывает «Все филиалы» / «Выбрано: N»
- Внутри: кнопки групп филиалов (если есть) + «Все филиалы» чекбокс + отдельные филиалы
- JS-функции: `onBranchCbChange()`, `toggleAllBranches()`, `selectBranchGroup(ids)`
- Параметр формы: `branch_ids` (multiple), принимается через `request.args.getlist('branch_ids')`
- Готовые примеры: `expenses_report.html`, `purchases.html`

### Выпадающий список контрагентов
- **С поиском внутри dropdown и множественным выбором галочками**
- Поле поиска фильтрует список контрагентов прямо внутри dropdown (client-side)
- JS-функции: `filterSuppliers()`, `onSuppCbChange()`, `toggleAllSuppliers()`
- Готовый пример: `purchases.html`

### Контрагенты — единая база
- Единая таблица `contractors` (раздел Банк) — источник истины для всех контрагентов/поставщиков

- Дублировать в `purchase_suppliers` для фильтров накладных
