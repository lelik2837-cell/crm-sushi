### 27. Исправление ложного матчинга контрагентов

**Проблема:** ООО "КОМПАНИЯ ВОДА" распознавалась как СОУСПАК, ООО
- Причина 1: текст для поиска включал `description` (назначение платежа) — слово «ООО» из ключевых слов СОУСПАК совпадало с «ООО» в поле description

- Матчинг теперь только против поля `counterparty`, не `description + counterparty`
- Если ключевые слова не заданы вручную (`keywords = ''`) → используется имя контрагента целиком как одна фраза (не разбивается по запятой)
- Если ключевые слова заданы (`keywords` заполнен) → разбиваются по запятой как прежде

### 112. Исправление часового пояса — дашборды показывали данные не того дня

**Проблема:** при пролистывании выручки на день назад «сегодня» показывало вчерашнюю выручку (и аналогично для произвольных дат) — из-за `Date.prototype.toISOString()`, которая конвертирует дату в UTC перед форматированием; в часовом поясе UTC+N это сдвигает дату на день назад в первые N часов суток.

**Исправление:** во всех местах, где дата бралась через `.toISOString().slice(0,10)`, добавлен локальный хелпер:
```javascript
function fmtLocalDate(d) {
    return d.getFullYear() + '-' + String(d.getMonth() + 1).padStart(2, '0') + '-' + String(d.getDate()).padStart(2, '0');
}
```
Заменено 9 мест использования в 3 файлах: `dashboard_owner.html` (оба `<script>`-блока — KPI и карточка выручки), `fot_dashboard.html`, `sber_settings.html`. Проверено через `grep -rn "toISOString"`, что в шаблонах не осталось других мест с этой же ошибкой.

Пустая строка в JS — тоже ложное значение, поэтому `!panel.style.display` после первого открытия снова давало `true` — функция каждый раз считала, что панель нужно ПОКАЗАТЬ, и никогда не переключала на `'none'`. После обновления страницы сервер заново рендерит `style="display:none"`, поэтому баг был незаметен до первого клика.

**Исправление:** проверка сужена до строгого `panel.style.display === 'none'`, а видимое состояние теперь ставится явным `'block'` вместо `''` — так оба значения однозначные, и клик корректно переключает туда-обратно.

**Проверено:** на тестовой копии базы страница смены с курьером отдаётся без ошибок, исправленная функция отрендерилась как ожидалось.

### 124. Исправление: скрипт синхронизации молча зависал на логине после деплоя

**Симптом:** после деплоя обновлённого `revenue_sync.py` на Railway в логах виден старт контейнера, сообщение о режиме опроса и предупреждение «CSRF-токен не найден» — и всё, тишина. Ни «Успешно авторизовались», ни ошибки. Данные не появлялись ни в вебхуках, ни в отчётах.

**Причина:** запросы к goulash внутри `get_csrf_token()` (GET страницы логина) и `login()` (POST формы логина) не имели `timeout` — в отличие от запроса выручки/времени ожидания и отправки на crmpapa.ru, где `timeout=15` уже был. Без таймаута `requests` ждёт ответ неограниченно долго; если сеть/сайт подвисли — процесс молча блокируется на этом месте навсегда, ничего не логируя (ошибка появилась бы, только когда/если запрос когда-нибудь завершится).

**Исправление:** добавлена общая константа `REQUEST_TIMEOUT` (30 сек по умолчанию, настраивается через `REQUEST_TIMEOUT_SECONDS`), применена ко всем трём запросам к goulash (`get_csrf_token`, `login`, `fetch_department_data`). **Симптом (сообщил пользователь):** во всех дашборд-отчётах (ФОТ, Рейтинги и т.д.) клик по кнопке группы филиалов ничего не делал.

**Причина:** в `_dash_branch_picker.html` (п.129) кнопка группы рендерилась как `onclick="dashBpSelectGroup('{{ bs_prefix }}', {{ g.branch_ids|tojson }}, {{ (g.abbr or g.name[:3].upper())|tojson }})"`. Фильтр `|tojson` для строки (сокращение группы) оборачивает значение в двойные кавычки (`"НК"`) — а сам атрибут `onclick="..."` тоже в двойных кавычках. В итоге браузер обрывал значение атрибута на первой попавшейся `"` из середины `tojson`-строки, `onclick` превращался в синтаксически неполный/битый JS, и обработчик клика либо не навешивался, либо падал молча — клик ничего не делал. `{{ g.branch_ids|tojson }}` (массив чисел) на эту же проблему не натыкался, поэтому баг проявлялся только на аргументе с текстом (сокращении группы), а не с id.

**Исправление:** значения группы (id-шники филиалов и сокращение) вынесены из `onclick`-строки в `data-branch-ids`/`data-abbr` — обычные Jinja-выражения `{{ }}` (без `|tojson`), которые Jinja сам корректно HTML-экранирует под атрибут независимо от содержимого. `onclick="dashBpSelectGroupFromBtn('{{ bs_prefix }}', this)"` — новая функция в `base.html` читает `this.dataset.branchIds`/`this.dataset.abbr` и вызывает существующий `dashBpSelectGroup(...)`.

**Проверено:** статически — атрибут `onclick` для всех 4 id-based дашбордов (Выручка/ФОТ/LFL/Рейтинги) теперь рендерится как валидный `onclick="dashBpSelectGroupFromBtn('fot', this)"` без утечки кавычек; поведенчески — прогнал реальный код `dashBpSelectGroup`/`dashBpSelectGroupFromBtn` через настоящий движок V8 (`py_mini_racer`) с эмуляцией DOM: симулированный клик по кнопке группы (`data-branch-ids="1,2,3,4"`, `data-abbr="НК"`) корректно выставляет `window.fotBranchIds=[1,2,3,4]`, текст кнопки → «НК», отмечает все 4 чекбокса филиалов, снимает чекбокс «Все филиалы» и вызывает `fotOnBranchChange` (перезагрузку карточки). Продовая база не затронута (баг чисто фронтенд).

### 139. Деплой — переезд с Railway на собственный сервер через GitHub Actions

Пользователь перенёс продакшен с Railway на свой сервер; деплой теперь по `push` в `main` — GitHub Actions (`.github/workflows/deploy.yml`) подключается по SSH (логин/пароль в GitHub Secrets) и выполняет `/opt/deploy.sh` на сервере.

**Исправление:** `git rebase origin/main` (конфликтов не было — коммиты правят разные файлы), затем `git push`. Первая попытка `push` упала — токен доступа (PAT), вшитый в URL `origin`, оказался отозван/истёк (`Invalid username or token. Password authentication is not supported`); пользователь сгенерировал новый classic PAT (права `repo`) в настройках своего GitHub-аккаунта, `origin` обновлён через `git remote set-url`, `push` прошёл. Статус самого деплоя на сервере проверен через GitHub Actions API (`GET /repos/.../actions/runs`, токен как Bearer) — `completed`/`success`.

**По прямой просьбе пользователя — записано в память (не в этот файл):** теперь после проверенных изменений в проекте нужно коммитить и пушить самостоятельно, с коротким осмысленным сообщением, не дожидаясь отдельной просьбы каждый раз — это отменяет более раннее указание от 2026-07-12 «не коммить самому», которое было привязано к старой схеме автодеплоя Railway. ### 158. Банк → Выписка / Выписка по всем — категории в фильтре показывались кодом, а не названием; найден и исправлен реальный корень «фильтр не работает» (скрипт вне `{% block scripts %}`)

Сразу после деплоя п.157 пользователь прислал скриншот и уточнил: категории в фильтре подписаны служебными кодами вроде `repair_electric` вместо человеческого названия, и если в дропдауне категории снять все галочки (список транзакций становится пустым/коротким), сам выпадающий список с чекбоксами не остаётся поверх таблицы, а прячется.

Первое — простой фикс: в `bank_statement_view` и `bank_statements_all` (`crm/app.py`) добавлен `cat_labels = {c['code']: c['label'] for c in get_expense_categories(conn)}` (по всем категориям, не только `show_contractors`, т.к. эффективная категория операции может быть любой), передаётся в шаблон; `unique_cats` теперь сортируется по названию (`key=lambda code: cat_labels.get(code, code)`), а в `bank_statement.html` чекбоксы показывают `{{ cat_labels.get(cat, cat) }}` вместо голого кода.

Второе оказалось куда серьёзнее и объясняло сразу оба симптома разом (и «список прячется», и общее ощущение «фильтр не работает»): весь `<script>` страницы физически находился **внутри `{% block content %}`**, отдельного `{% block scripts %}` в файле не было вовсе (в отличие от, например, `pnl.html`, где скрипты честно вынесены в `{% block scripts %}`). А в `base.html` `{% block content %}` (строка 540) стоит **до** тега `<script src=".../bootstrap.bundle.min.js">` (543), тогда как `{% block scripts %}` (627) — после. В итоге код страницы, инициализирующий кастомный Dropdown для фильтров категории/типа операции —
```js
new bootstrap.Dropdown(btn, { autoClose: 'outside', popperConfig: { strategy: 'fixed' } });
```
— выполнялся **раньше**, чем успевала загрузиться сама библиотека Bootstrap, и падал с `ReferenceError: bootstrap is not defined` при каждой загрузке страницы (не изредка — гарантированно всегда, порядок тегов в HTML детерминирован). ### 161. Простой P&L — исключение категории из выписки не влияло на сумму для операций, категоризированных только правилом разбора

### 165. «Сверка безнала» — «Приход банк» теперь берёт филиал из правила разбора операции, а не только из привязки счёта

Сразу после п.164 пользователь уточнил: в его случае филиал банковской операции проставляется именно через привязку в правиле разбора (то, что только что добавили), а не через привязку счёта к филиалу в «Банк → Счета» — и «Сверка безнала» (п.163) с ним не работала: категория в настройках выбрана, а колонка «Приход банк» пустая. Причина ожидаемая — при реализации п.163 «Сверка безнала» ещё не существовало привязки филиала на уровне правила (её добавили только что в п.164), поэтому запрос считал приход банка только через `JOIN bank_account_branches`, а у пользователя сам счёт к филиалу не привязан.

В `reconciliation_cashless()` (`crm/app.py`) блок расчёта «Приход банк» переписан: вместо одного SQL-запроса с `JOIN bank_account_branches` — построчная Python-обработка (SQL здесь в принципе не годится, т.к. совпадение правила по ключевому слову в описании делается в Python, как и везде в проекте, см. `_apply_bank_parse_rules`). Для каждой подходящей по категории/сумме/периоду транзакции: сначала проверяются активные правила разбора в том же порядке и с той же логикой совпадения (счёт/направление/ключевое слово), что и в `_apply_bank_parse_rules`; если у совпавшего правила указан филиал/группа (`bank_parse_rule_branches`, п.164) — используется он; если правило не совпало или совпало без привязки филиала — используется старое поведение, привязка самого счёта (`bank_account_branches`). Таким образом привязка через правило теперь имеет приоритет, а привязка через счёт остаётся резервным вариантом для тех, у кого правила не заведены. Фильтр по выбранным в отчёте филиалам применяется тоже в Python, после определения филиала транзакции (а не в SQL, как раньше), иначе фильтрация не учитывала бы переопределение по правилу.

**Проверено через Playwright** на копии `crm.db`: заведён банковский счёт БЕЗ привязки к филиалу в «Банк → Счета», правило разбора «Эквайринг КВАДРАТ» (ключевое слово «Эквайринг», направление «Приход») с привязкой к филиалу КВАДРАТ через `bank_parse_rule_branches`, транзакция прихода 12 000 ₽ с категорией, выбранной в настройках отчёта, и описанием, содержащим «Эквайринг» — до фикса такая транзакция нигде не появлялась (счёт без привязки), после фикса «Приход банк» на КВАДРАТ корректно показал 12 000 ₽, «Выручка безнал» из смены 13 000 ₽, разница +1 000 ₽. Проверен и фильтр отчёта по филиалам: при фильтре на чужой филиал (id=2) строка не показывается («Нет данных»), при фильтре на КВАДРАТ (id=1) — показывается. Тестовый сервер, копия БД и скрипт репродукции удалены после проверки.

### 166. «Сверка безнала» — фикс: категория тоже бралась только вручную, а не из правила (та же ошибка, что и филиал в п.165)

Пользователь сообщил, что после фикса п.165 «Приход банк» всё равно нулевой. Причина — соседний баг того же класса: SQL-фильтр `AND bt.category IN (?)` в `reconciliation_cashless()` фильтровал по **сырому** `bt.category`, а не по эффективной категории (`bt.category or parse_rule_category`), и применялся ДО того, как код вообще начинал проверять правила — то есть транзакции, у которых категория подставляется только правилом (ручное поле `category` пустое, как в реальном сценарии пользователя, где и категорию, и филиал определяет одно и то же правило), отсекались SQL-запросом ещё на входе, и Python-логика по правилам/филиалам (п.165) до них просто не доходила.

Исправлено: убран SQL-фильтр по категории из запроса — теперь выбираются все входящие транзакции периода, и категория проверяется в Python в той же самой петле, что уже проверяет правило ради филиала (нет смысла гонять её дважды): при переборе `parse_rules` дополнительно запоминается `rule.category`, эффективная категория считается как `bt.category or matched_rule_category`, и уже она сравнивается с выбранными в настройках отчёта категориями.

**Проверено:** на копии `crm.db` — банковский счёт без привязки к филиалу, транзакция с **пустым** `bt.category` (ничего не проставлено вручную), правило разбора с ключевым словом «Эквайринг», категорией `acq3` и привязкой к филиалу КВАДРАТ; в настройках отчёта выбрана категория `acq3`. До фикса такая транзакция не находилась вообще (SQL сразу отсекал по `bt.category=''`), после фикса «Приход банк» корректно показал 9 500 ₽ при выручке безналом 10 500 ₽ (разница +1 000). Тестовый сервер и копия БД удалены после проверки.

### 167. «Сверка безнала» — знак разницы перевёрнут, ссылка на смену слева от филиала

Пользователь попросил: 1) если приход банка меньше выручки безналом — разница должна быть минус, если больше — плюс (раньше было наоборот: считалось как «выручка минус банк», то есть недостача банка показывалась как плюс); 2) слева от названия филиала — сразу ссылка на смену за этот день.

В `reconciliation_cashless()` (`crm/app.py`) формула разницы поменяна с `card_revenue - bank_income` на `bank_income - card_revenue` — теперь недостача (банк < выручки) действительно отрицательная, излишек (банк > выручки) положительный. Вслед за этим в шаблоне поменяны местами условия подсветки цветом (`text-danger`/`text-warning`) на всех трёх уровнях (итоги периода, строка-итог дня, строка филиала): теперь красным подсвечивается именно недостача (`diff < 0`), а не то, что осталось от старого знака — иначе после переворота формулы получилось бы наоборот (излишек красным, недостача жёлтым), что не имело бы смысла.

Для ссылки на смену — в `card_rows` (запрос выручки безналом из `shifts`/`shift_revenue`) добавлено `s.id AS shift_id`; корректность обеспечена существующим `UNIQUE(branch_id, date)` на таблице `shifts` — на одну пару (филиал, дата) гарантированно не больше одной смены, так что `shift_id` однозначен и не требует доп. агрегации. В шаблоне `reconciliation_cashless.html` перед названием филиала в строке добавлена маленькая иконка-ссылка (`bi-box-arrow-up-right`, `target="_blank"`, `url_for('shift_view', shift_id=br.shift_id)`) — показывается только если для этой пары (дата, филиал) нашлась смена (для дней, где банк прислал деньги, а смены не было вообще — тот самый случай расхождения, ради которого и делалась сверка, — ссылки нет).

**Проверено через Playwright:** банк меньше выручки (8000 против 10000) — разница «-2 000 ₽» красным; после правки той же транзакции на 15000 (банк больше выручки) — «+5 000 ₽»; клик по иконке слева от «КВАДРАТ» открывает `/shift/970` в новой вкладке. Тестовый сервер, копия БД и скрипты репродукции удалены после проверки.
